# Human authorship and project control

MatchShift is a human-led hackathon project.

## Human project owner

The repository is owned and controlled by the GitHub account `bit-nexus-labs`.

The human project owner made and approved the material product decisions, including:

- entering the Consumer & Fan Experiences track;
- selecting the MatchShift spoiler-safe delayed-viewer concept;
- requiring a walletless, loginless, no-cost judge path;
- choosing server-side visibility enforcement instead of client-side score hiding;
- keeping deterministic synthetic replay as the guaranteed evaluation mode;
- limiting public TxLINE integration to documented backend snapshots and streams;
- excluding betting execution, wallets, rewards, settlement, and private trading-bot code;
- approving repository creation, branch strategy, pull requests, and merges;
- authorizing the implementation sequence and final review process.

The human owner remains responsible for:

- hackathon registration and eligibility;
- the truthfulness of submission claims;
- final review of the public demo and video;
- any TxLINE subscription or token activation;
- any wallet signature or transaction;
- final submission from the participant account.

## AI-assisted implementation

AI tools were used as implementation, research, review, testing, and documentation assistants under human direction.

AI tools did not:

- register as a participant;
- make an eligibility declaration;
- create or activate a TxLINE subscription on behalf of the participant;
- sign a wallet message or blockchain transaction;
- purchase a token or service;
- submit the project to the hackathon;
- replace the human participant's final decision-making authority.

## Development record

The Git history records incremental implementation through separate branches, pull requests, CI runs, and reviewed merges. Major milestones include:

1. spoiler-safe personal timeline core;
2. TxLINE snapshot/SSE adapter and official-schema review fixes;
3. judge-facing comparison UI;
4. production container and deployment runbook;
5. deterministic visibility receipts;
6. submission and compliance documentation.

## Final-submission requirement

Before submission, the human project owner should:

1. open the deployed judge URL in a private browser;
2. run the demo end to end;
3. review the repository README and submission text;
4. verify the video accurately reflects the product;
5. confirm no restricted branding, secrets, raw data exports, or misleading live-data claims appear;
6. submit personally through the official hackathon interface.
